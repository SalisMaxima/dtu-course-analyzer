Work on English language support for the DTU Course Analyzer.

## Current Implementation (v2.2.0)

Bilingual support is already implemented:
- 94.3% of courses have Danish name translations
- Language toggle in `extension/js/language-toggle.js`
- DataTables searches both hidden and visible columns

## Architecture

### Data Collection (`src/dtu_analyzer/scrapers/`)
- Danish: `BASE_URL/course/{id}?lang=da-DK` (**CRITICAL**: Must include `?lang=da-DK`)
- English: `BASE_URL/course/{id}?lang=en-GB`
- Without lang parameter, DTU returns English by default

### Data Structure (`coursedic.json`)
```json
{
  "01005": {
    "name": "Avanceret ingeniørmatematik",    // Danish
    "name_en": "Advanced Engineering Mathematics",  // English
    ...
  }
}
```

### Frontend (`extension/`)
- **Column visibility**: a `lang-da`/`lang-en` class on the `#course-table`
  element; CSS hides the other language's column. Both name columns always
  stay in the DOM, so rows rendered later follow the active language.
- **Toggle**: `extension/js/language-toggle.js` (external file for CSP)
- **Storage**: localStorage key `'dtu-analyzer-lang'`
- **Default**: English ('en')

## Key Files

- `src/dtu_analyzer/scrapers/async_scraper.py` — fetches both language versions
- `src/dtu_analyzer/analysis/analyzer.py` — generates bilingual db/data.json
- `extension/js/table.js` — renders the table, searches both name columns
- `extension/js/language-toggle.js` — toggle UI and column switching
- `extension/db.html` — table markup and the `lang-*` CSS rules

## NEVER DO

- ❌ Remove a name column from the DOM — the language toggle depends on both
- ❌ Use inline onclick handlers — CSP violation
- ❌ Forget `?lang=da-DK` parameter — returns English instead of Danish
